# 게스트 페이지 개편 구현 계획서

## 1. 구현 목표와 범위

1. 기존 `public/guest.html`의 레이아웃과 동작은 호환 대상으로 보지 않고 전면 교체한다.
2. 비로그인 사용자가 `https://imggul.com/{프로젝트 경로}`(예: `/hg`)로 접근하면 해당 프로젝트의 캐릭터와 관리자 게시글을 볼 수 있는 게스트 전용 화면을 제공한다.
3. 관리자 화면에는 프로젝트별 게시글 작성·조회·수정·삭제 기능을 추가한다.
4. 게스트는 이름과 비밀번호를 사용해 댓글을 작성하고, 자신이 설정한 비밀번호로 댓글을 수정·삭제할 수 있게 한다.
5. 기존 관리자 화면의 디자인 토큰, 헤더 높이, 색상, 다크 모드, 반응형 기준은 재사용하되 관리자 전용 상단 버튼인 `탐색기`, `이미지 생성`, `프로젝트`는 게스트 화면에서 렌더링하지 않는다.
6. 로그인 화면은 `/login`에서만 제공하고, 비로그인 사용자의 `/` 접근에는 별도의 잘못된 접근 화면을 반환한다.

## 2. 현재 구조와 변경 기준

1. 서버 라우팅과 API는 `functions/[[path]].js`의 Cloudflare Pages catch-all 함수가 담당하므로, 게스트 라우팅과 게시글·댓글 API도 이 파일에서 분기한다.
2. 현재 비로그인 `/`와 `/login`이 모두 `public/login.html`을 반환하는 분기를 분리한다. `/login`만 로그인 HTML을 반환하고 `/`는 신규 오류 화면을 반환하도록 수정한다.
3. 현재 `public/guest.html`은 오래된 탐색기형 화면이며 실제로 존재하지 않는 `/script.js`를 참조한다. 이 의존성을 제거하고 게스트 전용 ES 모듈을 새로 연결한다.
4. 프로젝트와 캐릭터의 기준 정보는 기존 D1의 `v2_projects`, `v2_characters`, `v2_assets` 및 현재 R2 폴더 구조를 재사용한다. 게시글과 댓글만 신규 D1 테이블로 분리한다.
5. 관리자 프로젝트 UI는 `public/js/project/shared.js`, `public/js/project/manage.js`, `public/js/project.js`의 기존 프로젝트 섹션·라우팅 방식을 확장한다.
6. 이미지 미리보기는 `public/guest.html`에 이미 있는 게스트용 `preview-modal`의 구조와 ID를 기준으로 정리해 재사용하되, 관리자용 편집·교체·삭제·공개 설정 동작은 포함하지 않는다.

## 3. URL 및 접근 제어

1. 게스트 프로젝트 URL은 첫 번째 경로 조각을 프로젝트 경로로 해석한다. `/hg`와 `/hg/`는 모두 D1의 `v2_projects.prefix = 'hg/'` 프로젝트로 해석하고, 화면 내 이동은 History API 또는 해시를 사용해 서버 프로젝트 경로와 충돌하지 않게 한다.
2. 경로는 `decodeURIComponent` 처리 후 정규화하고, 빈 값, `.`·`..`, 역슬래시, 중복 슬래시, 예약 경로(`api`, `login`, `logout`, 정적 자산 경로)는 프로젝트 경로로 인정하지 않는다.
3. 존재하지 않는 프로젝트는 HTTP 404 상태와 함께 "프로젝트를 찾을 수 없습니다" 화면을 제공한다. 프로젝트 경로 없이 접근한 `/`는 HTTP 404 또는 400 상태의 "잘못된 접근입니다" 화면을 제공한다.
4. `GET /login`은 비로그인 상태에서만 로그인 화면을 제공한다. 이미 로그인한 관리자가 접근하면 `/`의 관리자 화면으로 이동시킨다.
5. 로그인 성공 후에는 관리자 `/`로 이동하고, 로그아웃 후에는 잘못된 접근 화면이 아닌 `/login`으로 이동하도록 `/logout`의 리다이렉트 위치를 변경한다.
6. 관리자 인증이 필요한 API는 기존 `auth` HttpOnly 쿠키 검사를 유지한다. 게시글 쓰기 API와 관리자 댓글 삭제 API에는 반드시 `isAdmin` 검사를 적용한다.
7. 게스트 조회 API는 요청 URL의 프로젝트 경로를 서버에서 다시 프로젝트 ID로 해석한다. 클라이언트가 임의의 `projectId`, R2 prefix 또는 object key를 전달해 다른 프로젝트 데이터를 읽지 못하게 한다.

## 4. 게스트 페이지 레이아웃

1. `public/guest.html`을 관리자 `public/app.html`과 같은 기본 골격으로 교체한다. `h-[100dvh]`, 높이 14의 헤더, 동일한 색상·테두리·다크 모드·모바일 여백 규칙을 사용한다.
2. 헤더 왼쪽에는 ImgGul 로고, 앱 버전, 현재 프로젝트 이름을 표시하고 오른쪽에는 다크 모드 전환만 둔다.
3. 관리자 상단의 `탐색기`, `이미지 생성`, `프로젝트` 버튼과 로그아웃, 업로드, 편집용 UI는 HTML 단계에서 생성하지 않는다. CSS로 숨기는 방식에 의존하지 않는다.
4. 게스트 본문에는 `캐릭터`와 `게시글` 두 화면을 전환하는 게스트 전용 탭을 둔다. 초기 화면은 캐릭터 목록으로 하고 모바일에서도 두 탭이 한 줄에 안정적으로 표시되게 한다.
5. 로딩, 빈 목록, API 오류, 프로젝트 없음 상태를 각각 별도 상태 컴포넌트로 제공한다. 오류 메시지에는 서버 내부 경로나 예외 스택을 노출하지 않는다.
6. 신규 `public/js/guest.js`에서 프로젝트 초기화, 탭 전환, 목록 조회, 게시글 상세, 댓글 CRUD, 이미지 모달을 관리한다. 공통 다크 모드처럼 안전하게 재사용할 수 있는 함수만 기존 모듈에서 분리 또는 import한다.

## 5. 캐릭터 목록 및 이미지 조회

1. `GET /api/guest/projects/{projectPath}`를 추가해 프로젝트 이름, 경로, 캐릭터 요약 목록을 반환한다. 응답에는 프롬프트, 관리자 메타데이터, 비공개 작업 파일, 내부 R2 key 전체 목록을 포함하지 않는다.
2. 캐릭터는 기존 `v2_characters.sort_order` 순서로 정렬하고, 필요한 경우 기존 R2 캐릭터 폴더 목록을 보조 자료로 사용한다. D1과 R2가 불일치하면 서버가 허용된 이미지 확장자만 골라 반환한다.
3. 캐릭터 카드에는 캐릭터 이름, 경로 표시명, 대표 이미지, 이미지 개수를 표시한다. 대표 이미지는 첫 번째 유효 이미지 또는 명시된 대표 자산을 사용하고 없으면 공통 빈 이미지를 표시한다.
4. 캐릭터 선택 시 해당 캐릭터의 이미지 그리드를 조회한다. `webp`, `png`, `jpg`, `jpeg`만 허용하고 `_meta.json`, 로그, 임시 생성 폴더, 편집 세션 등의 내부 파일은 제외한다.
5. 게스트 이미지 요청은 프로젝트와 캐릭터 소속을 서버에서 확인하는 전용 조회 경로를 사용한다. 응답에는 적절한 `Content-Type`, ETag, 캐시 정책을 설정하되 권한 검사를 우회하는 임의 R2 key 조회는 허용하지 않는다.
6. 이미지 카드를 누르면 게스트용 미리보기 모달을 연다. 모달에는 이미지, 닫기 버튼, 간단한 프로젝트/캐릭터/파일 경로만 표시하고 관리자용 공개 설정, 편집, 교체, 이동, 삭제 기능은 넣지 않는다.
7. 모달은 배경 클릭, 닫기 버튼, `Escape` 키로 닫히게 하고 열릴 때 포커스 이동 및 닫힌 뒤 원래 카드로 포커스 복귀를 지원한다.

## 6. 게시글 데이터 모델과 마이그레이션

1. 다음 순번의 신규 D1 migration 파일을 추가해 `guest_posts` 테이블을 만든다. 필드는 `id`, `project_id`, `title`, `body`, `image_key`, `created_at`, `updated_at`을 기본으로 한다.
2. `guest_posts.project_id`는 `v2_projects.id`를 참조하고 프로젝트 삭제 시 게시글도 정리되도록 `ON DELETE CASCADE`를 설정한다. 목록 성능을 위해 `(project_id, created_at DESC)` 인덱스를 추가한다.
3. 같은 migration에 `guest_comments` 테이블을 만든다. 필드는 `id`, `post_id`, `author_name`, `body`, `password_hash`, `password_salt`, `created_at`, `updated_at`으로 구성한다.
4. `guest_comments.post_id`는 `guest_posts.id`를 참조하고 게시글 삭제 시 댓글도 삭제되도록 한다. `(post_id, created_at ASC)` 인덱스를 추가한다.
5. 모든 시간은 기존 `nowIso()` 정책과 같은 서버 생성 KST ISO 문자열(`+09:00`)로 저장한다. 화면에서는 `Intl.DateTimeFormat('ko-KR')`로 표시하며 클라이언트가 보낸 작성 시간을 신뢰하지 않는다.
6. 게시글 이미지는 R2의 프로젝트 하위 전용 prefix(예: `{projectPrefix}_guest_posts/{postId}/...`)에 저장하고 D1에는 object key만 기록한다. 게시글 수정으로 이미지가 교체되거나 게시글이 삭제되면 이전 R2 객체도 정리한다.
7. DB 저장 실패 시 먼저 업로드된 R2 객체를 정리하고, R2 삭제 실패 시에는 게시글 삭제 자체는 완료한 뒤 재정리 가능한 로그를 남기는 방식으로 부분 실패를 처리한다.

## 7. 관리자 게시글 작성 화면

1. `public/js/project/shared.js`의 `PROJECT_SECTIONS`에 `게시글` 섹션을 추가하고, `public/js/project/manage.js`의 프로젝트 대시보드에 게시글 카드를 배치한다. 기존 카드 레이아웃은 6개 섹션에서도 데스크톱과 모바일 비율이 깨지지 않도록 재조정한다.
2. `public/js/project/posts.js`를 신규 작성해 게시글 목록, 작성 폼, 수정 폼, 삭제 확인, 이미지 미리보기를 담당하게 하고 `public/js/project.js`에서 export하여 기존 `window` 바인딩 방식과 연결한다.
3. 게시글 작성 폼은 제목, 본문, 이미지 1개(선택)를 지원한다. 이미지 제거와 교체를 제공하고 저장 전에 파일 형식과 크기를 클라이언트에서도 1차 검사한다.
4. 관리자 API는 `POST /api/admin/projects/{projectId}/posts`, `PATCH /api/admin/posts/{postId}`, `DELETE /api/admin/posts/{postId}`로 분리한다. 모든 요청에서 관리자 쿠키와 게시글의 프로젝트 소속을 서버에서 검증한다.
5. 게시글 목록 및 편집 화면에는 작성 시간과 마지막 수정 시간을 표시한다. 수정된 글은 `수정됨` 표시를 추가하고 실제 `updated_at`을 함께 보여준다.
6. 삭제는 확인 대화상자를 거친 뒤 수행하고, 성공 후 프로젝트 게시글 목록과 게스트 조회 캐시가 즉시 갱신되게 한다.
7. 제목·본문 길이, 이미지 MIME type, 확장자, 최대 바이트 수를 서버에서도 검증한다. HTML은 저장하거나 직접 렌더링하지 않고 일반 텍스트로 취급해 스크립트 삽입을 방지한다.

## 8. 게스트 게시글 조회 화면

1. `GET /api/guest/projects/{projectPath}/posts`에서 최신 작성 순으로 게시글 요약을 반환하고, 필요하면 cursor 기반 페이지네이션을 적용한다.
2. `GET /api/guest/projects/{projectPath}/posts/{postId}`는 본문, 이미지 URL, 작성·수정 시간, 댓글 목록을 반환한다. URL의 프로젝트와 게시글의 `project_id`가 일치하지 않으면 404로 처리한다.
3. 게시글 목록 카드는 제목, 본문 일부, 썸네일, 작성 시간, 댓글 수를 표시한다. 상세 화면에서는 전체 본문, 원본 비율 이미지, 수정 여부, 댓글을 표시한다.
4. 게시글 이미지도 프로젝트 소속을 확인하는 전용 자산 응답을 사용하며, 이미지 클릭 시 캐릭터 이미지와 동일한 게스트 미리보기 모달을 재사용한다.
5. 목록으로 돌아갈 때 스크롤 위치를 보존하고 새로고침이나 직접 링크 진입 시에도 동일한 게시글 상세를 복원할 수 있도록 게스트 내부 route state를 관리한다.
6. 게시글이 없을 때는 "등록된 게시글이 없습니다" 상태를 표시하며 캐릭터 조회 기능은 정상적으로 계속 사용할 수 있게 한다.

## 9. 댓글 작성·수정·삭제

1. 게스트 댓글 작성 폼은 이름, 비밀번호, 내용 필드를 제공한다. 비밀번호는 입력 시 마스킹하고 저장 완료 후 DOM과 메모리에서 지운다.
2. `POST /api/guest/projects/{projectPath}/posts/{postId}/comments`로 댓글을 작성한다. 서버는 프로젝트·게시글 소속, 이름·본문 길이, 비밀번호 최소·최대 길이를 검증한다.
3. 댓글 비밀번호는 평문이나 복호화 가능한 형태로 저장하지 않는다. Cloudflare Workers Web Crypto의 PBKDF2와 댓글별 무작위 salt를 사용해 hash를 만들고 `password_hash`, `password_salt`만 D1에 저장한다.
4. `PATCH /api/guest/comments/{commentId}`와 `DELETE /api/guest/comments/{commentId}`는 요청 본문의 비밀번호를 기존 hash와 constant-time 방식으로 비교한 뒤 처리한다. 수정 시 이름과 본문을 변경할 수 있게 한다.
5. 댓글에는 작성 시간과 수정 시간을 표시하고 수정된 댓글에는 `수정됨` 표기를 추가한다. 댓글 ID, 이름, 본문, 시간 외에 hash·salt·내부 프로젝트 정보는 응답하지 않는다.
6. 관리자는 관리자 인증 쿠키가 있으면 댓글 비밀번호 없이 삭제할 수 있게 한다. 관리자 게시글 상세 화면에 댓글 삭제 버튼을 제공하고 같은 `DELETE` 처리에서 `isAdmin`이면 비밀번호 검사를 생략한다.
7. 댓글 수정·삭제 실패 시 비밀번호 오류와 존재하지 않는 댓글을 사용자에게 구분 가능한 수준으로 안내하되 hash 정보나 내부 쿼리 결과는 노출하지 않는다.
8. 댓글 작성 API에는 본문 크기 제한, 요청 빈도 제한 또는 최소한의 IP 기반 단기 throttling, 동일 요청 중복 방지, JSON Content-Type 검사를 적용해 기본적인 도배와 과도한 요청을 줄인다.
9. 화면 출력은 `textContent` 또는 기존 `escapeHtml` 유틸을 사용하고 줄바꿈만 안전하게 표현한다. 사용자 입력을 `innerHTML`에 그대로 삽입하지 않는다.

## 10. API 응답과 오류 처리 원칙

1. 성공 응답은 `{ data: ... }`, 오류 응답은 `{ error: { code, message } }`처럼 일관된 JSON 형태로 맞춘다.
2. 입력 오류는 400, 인증 실패는 401, 관리자 권한 부족은 403, 프로젝트·게시글·댓글 없음은 404, 댓글 비밀번호 불일치는 403, 중복 또는 상태 충돌은 409로 구분한다.
3. 목록 API는 필요한 필드만 `SELECT`하고 비밀번호 관련 열은 어떠한 응답 경로에서도 선택하거나 직렬화하지 않는다.
4. 게시글 또는 댓글 갱신 쿼리는 `id`와 소속 ID를 함께 조건에 넣어 다른 프로젝트 객체를 수정하지 못하게 한다.
5. 서버 로그에는 요청 실패 위치와 식별 가능한 record ID만 남기고 댓글 비밀번호, 관리자 secret, 쿠키, 게시글 본문 전체는 기록하지 않는다.
6. 공개 API와 관리자 API의 캐시 정책을 분리한다. 관리자 및 댓글 변경 응답은 `no-store`, 게스트 목록은 짧은 캐시 또는 `no-cache`와 ETag를 적용하고 변경 후 오래된 내용이 남지 않게 한다.

## 11. 파일별 수정 계획

1. `functions/[[path]].js`: 로그인·루트·프로젝트 경로 라우팅 분리, 프로젝트 resolver, 게스트 조회 API, 관리자 게시글 API, 댓글 인증 및 CRUD, 게시글 이미지 R2 처리를 추가한다.
2. `migrations/{다음 순번}_guest_posts_comments.sql`: `guest_posts`, `guest_comments`, 외래 키와 조회 인덱스를 추가한다.
3. `public/guest.html`: 기존 탐색기 레이아웃을 삭제하고 관리자 디자인과 통일된 게스트 헤더, 캐릭터·게시글 탭, 상태 화면, 게스트 이미지 모달의 정적 골격으로 교체한다.
4. `public/js/guest.js`: 게스트 프로젝트 초기화, 캐릭터·이미지·게시글·댓글 렌더링, API 호출, route state, 모달 및 폼 상태를 구현한다.
5. `public/login.html`: 로그인 성공 이동과 오류 표시를 새 라우팅 정책에 맞게 확인하고, 필요한 경우 이미 로그인한 사용자의 처리만 보완한다.
6. `public/app.html`: 관리자 프로젝트 게시글 화면에서 공통으로 필요한 컨테이너나 모달이 있을 때만 최소 수정한다. 게스트 UI를 이 파일에 조건부로 섞지 않는다.
7. `public/js/project/shared.js`: `게시글` 프로젝트 섹션 정의와 공통 표시 유틸을 추가한다.
8. `public/js/project/manage.js`: 프로젝트 대시보드 게시글 카드, 섹션 진입, 6개 카드 반응형 레이아웃을 추가한다.
9. `public/js/project/posts.js`: 관리자 게시글 및 댓글 관리 화면을 신규 모듈로 구현한다.
10. `public/js/project.js`: 신규 게시글 모듈을 export하고 프로젝트 route 복원 흐름에 게시글 목록·상세 상태를 연결한다.
11. `public/style.css`: 관리자·게스트에서 함께 사용할 수 있는 최소 공통 스타일만 추가하며, 가능한 범위에서는 기존 Tailwind utility 구성을 유지한다.

## 13. 검증 계획과 완료 조건

1. 로컬에서는 프로젝트 규칙에 따라 브라우저 E2E나 라이브 서비스 테스트를 실행하지 않고, `node --check`를 이용한 JavaScript 문법 검사, import 경로 확인, migration SQL 검토, BOM 검사를 수행한다.
2. 수정한 모든 텍스트 파일이 UTF-8 without BOM인지 확인하고 시작 바이트가 `EF BB BF`가 아닌지 검사한다.
3. 사용자가 변경사항을 commit·push한 뒤 Cloudflare 배포 서버에서 `/login`만 로그인 화면을 표시하고, 비로그인 `/`는 잘못된 접근 화면을 표시하는지 확인한다.
4. 배포 서버에서 유효한 `/hg`, trailing slash가 있는 `/hg/`, 없는 프로젝트, 예약 경로를 각각 확인한다.
5. 캐릭터 정렬, 빈 캐릭터, 이미지 없는 캐릭터, 여러 확장자 이미지, 내부 메타·임시 파일 제외, 게스트 모달의 경로 표시와 닫기 동작을 확인한다.
6. 관리자 게시글의 글만 작성, 이미지만 포함, 글과 이미지 포함, 이미지 교체, 이미지 제거, 수정, 삭제를 확인하고 각 경우 R2 잔여 파일도 확인한다.
7. 게스트 댓글 작성 후 올바른 비밀번호로 수정·삭제되고 잘못된 비밀번호로는 거절되는지 확인한다. 관리자 쿠키 상태에서는 비밀번호 없이 댓글 삭제가 되는지도 확인한다.
8. 게시글과 댓글의 작성·수정 시간이 한국 시간 기준으로 일관되게 표시되고, 새로고침 후에도 동일하게 유지되는지 확인한다.
9. 제목·본문·이름에 HTML과 스크립트 문자열을 입력해도 실행되지 않고 텍스트로 표시되는지 확인한다.
10. 다른 프로젝트의 post ID, comment ID, asset key를 조합한 요청이 404 또는 403으로 차단되는지 확인한다.
11. 모바일과 데스크톱에서 관리자와 동일한 시각 체계를 유지하면서 게스트 헤더에 관리자 전용 3개 버튼이 생성되지 않는지 확인한다.
12. 위 항목이 모두 통과하면 기존 게스트 탐색기 코드와 더 이상 사용하지 않는 이벤트·스타일을 제거하고 기능 구현을 완료한 것으로 본다.

## 14. 구현 시 확정할 기본 제한값

1. 초기 구현값으로 게시글 제목 100자, 본문 10,000자, 댓글 이름 30자, 댓글 본문 2,000자, 댓글 비밀번호 4~72자를 적용한다.
2. 게시글 이미지는 1개, `image/jpeg`, `image/png`, `image/webp`만 허용하고 최대 10MB로 제한한다.
3. 게시글 목록은 페이지당 20개, 댓글은 기본 작성 순 전체 조회로 시작하되 댓글이 많아질 경우 cursor 페이지네이션을 추가할 수 있게 API 형태를 설계한다.
4. 제한값은 서버 상수로 한곳에서 관리하고 클라이언트는 안내와 사전 검증에 같은 값을 사용한다. 실제 구현 전에 운영 정책이 다르면 이 값만 조정한다.
